# Installed base — Colorado Environmental Data Hub

This file renders the CUPIDS Lab values spine as concrete requirements for **this** repository. Public-interest data infrastructure sits under three chronic pressures; we answer each with a counter-value the scaffold has to operationalize, and we hold six installed-base elements as things the repo must contain — not aspirations. Read `responsible-data-checklist.md` and `GOVERNANCE.md` alongside this; the point of the spine is to keep affordances coupled to the duties that make them contestable. This project is unusually on-point for the spine: its entire premise is that Colorado environmental data is **eroding** and **enclosing**, so the counter-values below are also the project's subject matter.

## Three pressures, three counter-values

- **Enclosure → Openness.** Public evidence keeps getting locked into formats and platforms that are technically open but practically closed — the catalog flags this directly with an `enclosure` field per source. This repo must keep its own data and pipelines in durable, legible, version-controllable formats (JSON, CSV, Markdown, text-based pipeline definitions), make everything a hosted tool produces (Datasette, R2, Quarto) exportable, and avoid taking on any vendor-locked artifact as a required dependency — because lock-in is itself a form of enclosure.
- **Exemption → Oversight.** Powerful actors carve out exemptions from scrutiny. This repo makes oversight routinized and testable: per-row `run_id` provenance, reviewable change logs and version history, the monthly CI refresh, and a named role (`ROLES.md`) authorized to interpret and contest the evidence — including the future corpus's coding labels.
- **Erosion → Ownership.** These projects quietly erode when grants, pilots, semesters, or administrations end — and so does the upstream data the project tracks (the catalog's `erosion` flag). This repo carries stewardship obligations that survive turnover: named custodians (CUPIDS Lab), a retention schedule, archived snapshots in R2, citable Dataverse deposits, and reliability checks re-run each semester — so the default trajectory is continuity rather than abandonment.

## Six installed-base elements, as repo requirements

1. **Jurisdictional linkability** — this repo must use stable, persistent identifiers and document the crosswalks that let records join across sources, decisions, and time, and must name who owns the reference data. Today: stable source IDs in the catalog and per-site keys in the reservoir dataset. Future: the journalist → article → data-source crosswalk that is the corpus's whole point, with archived URLs as the durable join key. (Lives in the relevant `DATA-DICTIONARY.md`.)
2. **Provenance-in-practice** — this repo must keep provenance, change logs, and version history that are expected and reviewable, and legible to affected parties rather than only to builders. (Provenance fields + per-extract `run_id` in `DATA-DICTIONARY.md`, plus `decision-log.md` and `CHANGELOG.md`.)
3. **Stewardship continuity** — this repo must record retention schedules, redundancy/mirroring, a migration plan, and named custodians who outlast budget cycles. (`CHARTER.md` "what survives the pilot"; the retention section of `GOVERNANCE.md` and `data-management-plan.md`; monthly CI rebuild + Harvard Dataverse archive + R2 redundancy.)
4. **Legitimate visibility / tiered access** — this repo must state documented decision rules for what is public, what is restricted, and which accountable intermediaries may touch sensitive traces. (Access tiers in `GOVERNANCE.md`: public catalog/datasets, restricted licensed content, the provisioned-but-empty sensitive tier.)
5. **Contestable oversight** — this repo must define logging and audit rights, a reporting cadence, and a role authorized to interpret and contest the evidence. (`ROLES.md` plus `GOVERNANCE.md`; the subject-matter expert contests coding labels, the project manager holds audit rights.)
6. **Remedy / escalation** — this repo must provide an accessible, protective, consequential pathway that turns a documented problem into review and correction, with transparency about the outcome. (The escalation section of `GOVERNANCE.md`, including the journalist correction / right-to-respond pathway.)

## How this scales

This spine is not a license to over-build. For the **public catalog the repo ships today**, most of these elements collapse to a sentence in the `README.md` and a line or two here — the data is open government metadata and the duties are light. The weight scales up sharply with the **future news corpus**, where sensitivity, copyright, and journalist privacy enter. The one place sensitivity overrides leanness is the coupling below, where the duty ships with the affordance no matter how small the project.

## Minimum viable safeguards (the load-bearing coupling)

This project will handle **conditionally sensitive-human** data once the corpus ingests article text + journalist records, so this section is non-negotiable *for that build*. Identifiers and retention are not neutral goods. Every linkability and retention affordance the corpus introduces — a journalist identifier, an article-to-source crosswalk, a preserved HTML snapshot, an access log — must be paired with the access tiers and contestable governance in `GOVERNANCE.md` over what is linkable, what is preserved, who can reach sensitive traces, and how those decisions are challenged. Concretely: this repo must not introduce a journalist-identifier scheme, a retention extension, or new logging without also landing the matching access-tier and remedy/escalation language in `GOVERNANCE.md`, and must not ingest article text or journalist records until the **blocking** items in `ROADMAP.md` are satisfied. Affordances without duties are theater, and this repo refuses to ship the theater. (No Indigenous data is implicated today; apply CARE if a source in scope ever changes that.)
