# Decision log — Colorado Environmental Data Hub

A running, dated record of design and data decisions and *why* they were made — the provenance of the project's choices, legible to people who join later. Add an entry whenever you make a non-obvious choice (a filtering rule, a definition, a tool, a tradeoff). Newest first.

## 2026-06-23 — Relocate the source catalog to `docs/planning/`; merge `retro/` into `audits/`; retire the top-level `data/` scaffold

- **Context:** the repo was scaffolded with a generic top-level `data/{raw,interim,processed,external}` tree, but every real dataset lives under `pipelines/<name>/data/{original,processed,audit,lookups}` (different names entirely). The only thing top-level `data/` ever held was the curated catalog `colorado_environmental_data_sources.json` — itself **byte-identical** to a copy kept in `context/` — so the scaffold was empty, misleading, and double-stored the one asset. Separately, project records were split across two directories (`audits/` and `retro/`) with inconsistent front-matter (the 2026-06-22 audit had none).
- **Decision:** move the catalog to **`docs/planning/colorado_environmental_data_sources.json`** (it is a curated planning input the future corpus pipeline consumes — `context/architecture.md` nb-04), **dedupe** the `context/` copy to a single canonical file, **delete** the top-level `data/` tree and its `.gitignore` carve-out, and **reframe** the data-model docs (`AGENTS.md`, `CONTRIBUTING.md`, the PR template, `.skills/data-intake/SKILL.md`, `data-management-plan.md`) from the unused top-level model to the pipeline-centric reality. Merge `retro/` into **`audits/`** with subdirectories by document type — `audits/qa/` (`doc_type: audit`), `audits/after-action/` (`after-action-report`), `audits/revision-plans/` (`revision-plan`) — and enforce one YAML front-matter + `YYYY-MM-DD-slug.md` convention across all nine.
- **Why:** the scaffold described a layout the project doesn't use; keeping it (and a duplicated catalog, and two homes for the same kind of record) invited drift and confused new contributors about where things live. The immutability ethos is unchanged — it just lives where the data actually is (each pipeline's `data/original`/`data/lookups`).
- **Consequences:** there is **no top-level `data/`**; the catalog has one canonical home beside the planning docs (referenced by `DATA-DICTIONARY.md`, `GOVERNANCE.md`, the README); all point-in-time project records live under `audits/<type>/`. Supersedes the 2026-06-17 "immutable raw in `data/raw/`" decision below.

## 2026-06-23 — Extract a shared `co_pipeline_core`; auto-discover the CI matrix (#54, #55)

- **Context:** the four liberation pipelines (`reservoir`, `streamflow`, `snowpack`, `climate_stations`) had been stamped out from one another, leaving ~49% copy-paste duplication — `provenance.py` was AST-identical ×4, and the fetch engine / clean orchestration / `normalize_long` / audit profiling were near-identical (AAR `audits/after-action/2026-06-23-pipeline-deployments-aar.md` §4.1). Each new pipeline also needed a hand-edited CI matrix row, which had already broken `main` once (PR #47, AAR §4.3).
- **Decision:** create a shared, domain-agnostic package **`pipelines/_core/` (`co-pipeline-core`)** and extract the plumbing once — `provenance`, `config`, `schema.normalize_long`, the `fetch` engine, `clean` orchestration, and `audit` profiling — landing **one module per PR** (#60–#65), each proven on one pipeline before rolling to the rest. Each pipeline now depends on it via a uv path dependency (`{ path = "../_core", editable = true }`) and keeps only its domain logic (`sources`, `parsers/`, `stations`, the `schema` column vocabulary, `audit` extensions). The `pipeline` CLI skeleton is **kept per-pipeline** (retrieve→publish orchestration is domain-specific). Separately (#55), both CI workflows now **auto-discover** their matrix from `pipelines/*/pyproject.toml` via `.github/scripts/discover-pipelines.sh` (the `_core` library is excluded; a README `N tests` drift fails the run).
- **Why:** at four-plus pipelines the maintenance cost of N copies (one bug fixed N times) outran the write-time convenience of stamping; one provenance / one fetch engine is the structural payoff. The coupling a shared core introduces (a change can break all four) is mitigated by the `_core` test suite plus the auto-discovered matrix running every pipeline on every `_core` change. Below ~3 pipelines copy-paste was the defensible call — this is the documented threshold fed back into the `data-project` skill (`audits/revision-plans/2026-06-23-data-project-skill-revision-plan.md`).
- **Consequences:** `pipelines/_core/` ships its own test suite in its own CI job (outside the pipeline matrix); per-pipeline `src/` shrinks to domain logic. Adding a 5th pipeline is now: stamp a thin domain package against the shared core, drop in a `pyproject.toml` — no workflow edit. The per-pipeline QA audit that followed (`audits/qa/2026-06-23-pipelines-qa.md`, #56) verified all three post-reservoir pipelines run green on the shared core. See `pipelines/_core/README.md`.

## 2026-06-23 — Snowpack pipeline (#11): NRCS SNOTEL + snow courses, two networks not two agencies

- **Context:** #11 names NRCS SNOTEL/AWDB as the primary snowpack source, *and* "CO DWR/CDSS climate stations" + CoAgMet. But the later sibling #44 carves the **CDSS climate-station** domain (12k+ stations, `SnowSWE`/`SnowDepth`) into its own pipeline and asks #11 to focus on NRCS with a SNOTEL↔CDSS crosswalk. Built `pipelines/snowpack/` stamped from `streamflow` (#10).
- **Decision:** scope the pipeline to the **NRCS AWDB REST API** only, as **two sources/networks**: `nrcs_snotel` (`SNTL`, automated DAILY WTEQ/SNWD/PREC, ~1980+) and `nrcs_snowcourse` (`SNOW`, manual SEMIMONTHLY WTEQ/SNWD, back to the 1930s–40s). Defer CDSS climate snow to #44; document the crosswalk discipline in `concepts.yaml` and provide `audit.colocated_pairs` as the matching mechanism. CoAgMet out of scope. Ship **all 199 active stations** as the seed (full-state coverage, not a curated sample — SNOTEL/snow courses are a small, enumerable universe, unlike #44's 12k). The signature product is **basin percent of normal** (`audit.basin_percent_normal`, sum-of-stations vs the 1991–2020 day-of-year median), reconciled against the NRCS basin report.
- **Why:** respecting the #11/#44 boundary keeps each pipeline single-responsibility and avoids double-owning the CDSS snow data. SNOTEL + snow courses from one API is genuinely "multi-source from day one" (mirroring streamflow) while giving the deep pre-1980 history #11 wants. Unlike streamflow's USGS↔DWR **re-serve** (same gage, identical values), SNOTEL and snow courses are **nearby-not-identical** sites — so the cross-source check is a *looser* co-location correlation, not an identity, and the two are kept as complementary series (never averaged). AWDB needs **no API key** (simpler than the CDSS pipelines).
- **Consequences:** `data/lookups/sites.csv` is regenerable via `scripts/build_sites_seed.py` (`--all` for discontinued stations / even deeper history). Temperature is left out (the documented SNOTEL extended-range temperature-sensor bias) but declared in the vocabulary with the caveat. `precip_accum_in` is water-year-cumulative — documented as a difference-not-sum trap. 28 tests pass offline; a live Gunnison sample validated end-to-end (406k rows to 1936; Gunnison 106% of normal on 2024-04-01). CI matrix + Dataverse kit added like the siblings; first human-reviewed publish remains gated behind the L4 QA checklist (#36). See `pipelines/snowpack/AGENTS.md`.
## 2026-06-23 — Cut label clutter: priority / size / level become Project fields

- **Context:** issues carried 4–6 label chips (a typical one: priority + size + level + type + blocking), which overwhelms a human scanning the list. priority/size/level were *also* GitHub Project single-select fields, so the planning metadata was double-encoded — and the board's Priority/Size/Level fields were in fact **empty**, because the sync set them with an invalid `gh` syntax that silently no-op'd.
- **Decision:** keep only **`type:*`, `blocking`, `good-first-issue`, `help-wanted`** as labels; move **priority, size, level** into the Project **fields** of the same name; delete GitHub's nine unused default labels (incl. the space-duplicates `good first issue` / `help wanted`). 28 labels → **8**; ~5 chips/issue → **2**. Populated the fields for all 20 board items from the existing labels first, then deleted the labels. Reworked `seed-github.sh` to set fields by GraphQL node id (`project_set_select`) from explicit per-task values, and to set `Status=Todo` only on issue *creation* so re-syncs never reset a moved card. Pruned `labels.yml`; updated `PROJECT-MANAGEMENT.md`.
- **Why:** a label should answer "what kind of thing is this," not carry planning dimensions the board already sorts by — duplication invites drift and buries the signal. Fixing the silently-broken field sync was a prerequisite (the board can now actually group by Priority/Size/Level).
- **Consequences:** `ROADMAP.md`'s Priority/Size/Level columns stay the human source of truth; the sync projects them onto the Project fields. Do not re-add priority/size/level as labels (noted in `labels.yml`). Re-running `seed-github.sh` stays idempotent and keeps the fields populated.

## 2026-06-22 — Climb to L4 (collaboration + responsible-data & accessibility)

- **Context:** the Hub had stayed at L1 by design while the reservoir-storage pipeline was built (L2 lives inside `pipelines/<name>/`). Two pressures made the climb due: the **undergraduate research team is about to be onboarded** (which the roadmap flagged as *blocking before onboarding*), and the **news corpus** — the project's flagship next build — carries copyright/fair-use and journalist-privacy duties that were only hand-encoded as roadmap prose. Asked to elevate the project to L3 or L4.
- **Decision:** climb straight to **L4** (which includes L3). Generated the L3 collaboration scaffolds (`CONTRIBUTING`, `CODE_OF_CONDUCT`, `ROLES` + `.github/CODEOWNERS`, `GOVERNANCE`, `CHARTER`, `collaboration-protocol`, `contributed-data-intake`, nested `.skills/`) and the L4 responsible-data & accessibility layer (`INSTALLED-BASE`, `data-management-plan`, `responsible-data-checklist`, `data-bulletproofing-checklist`, `data-quality-checklist`, `accessibility-checklist`, `data-card`). Filled each with real project content from `context/methodology.md` rather than shipping generic scaffolding. Repurposed `contributed-data-intake.md` for the project's actual external-data case — the **licensed-database manual-export workflow** (Nexis Uni/NewsBank/ProQuest/Factiva), not public crowdsourcing. Set the `GOVERNANCE.md` **sensitive tier as provisioned-but-empty**: the data today is public, so the duty is written down *before* the affordance exists.
- **Why:** L4 was chosen over a leaner L3-plus because the responsible-data duties were **already fully specified** in `context/methodology.md` (archive-every-URL, α ≥ 0.80, LLM quote-span guard, metadata-not-full-text, correction/right-to-respond), so the checklists fill with authored content, not boilerplate — and the project's own retro flagged "conditional sensitivity, hand-encoded as blocking ROADMAP rows" as a gap that climbing to L4 closes. Doing L3 at the same time is the real present need: it's what the team is onboarding into.
- **Consequences:** the previously-deferred L3/L4 concerns moved out of the roadmap's "deferred" list — the scaffolds are now built, and the *operationalization* work (name the CEJ contact, secure a fair-use/ToS review, fill the codebook + calibration, validate the LLM detector, run the QA gates before the first Dataverse publish) became **tracked, mostly-blocking** issues #31–#36 under a new `L3-L4: collaboration & responsible-data` milestone. `ROADMAP.md`'s deferred list is now essentially **L5** (OKF bundle, Datasette + Quarto site, agent-discovery, zero-copy sharing). Scaffolded on a branch + PR (shared repo), not committed to `main`.

## 2026-06-19 — Monthly CI/CD refresh + Harvard Dataverse as the dataset's home of record

- **Context:** the reservoir-storage pipeline produces a tidy CSV that needs re-deriving as upstream telemetry updates, a durable home between (ephemeral) runs, and a citable archive. Asked to automate a monthly refresh and to integrate the Dataverse deposit that landed in `data-project-skill` [PR #7](https://github.com/CUPIDS-Lab/data-project-skill/pull/7).
- **Decision:** schedule the refresh with **GitHub Actions** (monthly cron, matrix-per-pipeline) running a **full rebuild** — the composite-key de-dup in `clean.run` makes a rebuild a superset-append that also self-heals upstream revisions. Adopt **Harvard Dataverse** as the canonical home (citable DOI) via the skill's deposit kit, rendered into `pipelines/reservoir-storage/dataverse/`. CI **validates** the kit every run and deposits only a **draft** (opt-in `DATAVERSE_DEPOSIT`), **never** minting/publishing a DOI unattended.
- **Why:** a monthly batch doesn't justify a long-running Prefect agent; Actions is native, free, and secret-aware. Full rebuild is the simplest correct thing (and needs no restored prior CSV). Dataverse gives FAIR, citable archiving; the skill's hard rule — a deposit stops at a draft, publish only on explicit confirmation, since a DOI is permanent — is preserved by keeping CI draft-only and human-gated.
- **Consequences:** the first deposit creates the dataset (human reviews + publishes v1.0); recurring monthly updates should target that existing dataset (a new version, not a new DOI) via the `data-project-depositor` idempotent workflow — until configured, `DATAVERSE_DEPOSIT` stays off so the job validates without draft clutter. Needs repo config: `DATAVERSE_COLLECTION` (real alias), `DATAVERSE_API_TOKEN` (secret), optional `CDSS_API_KEY`. Streamflow (#10) / snowpack (#11) inherit the pattern with a one-line matrix add + their own `dataverse/` kit.

## 2026-06-19 — First data-liberation pipeline (reservoir storage) under `pipelines/`

- **Context:** issue #9 (reservoir storage) was the first slice of the water-liberation epic (#7) to be built, using the `data-liberation` skill.
- **Decision:** scaffold it as a **self-contained pipeline under `pipelines/reservoir-storage/`** — thin notebooks over a tested `src/reservoir/` package — establishing the **`pipelines/<name>/` convention** for liberated datasets. Sources: CO DWR/CDSS + USBR Reclamation RISE (Northern Water resolved as **boundaries-only**; its C-BT reservoirs are sourced from RISE). Scope filtered to **118 major reservoirs** (recharge/ag ponds dropped). `data/original/` is treated as a **rebuildable cache**, not an archive. (The monthly CI refresh + Harvard Dataverse deposit added on top are recorded in the entry above.)
- **Why:** the Hub will host several liberated datasets (streamflow, snowpack, …); one pipeline per directory keeps them independent and individually testable. `src/<pkg>/` (over the data-liberation template's `scripts/`) matches the repo's package convention and keeps notebooks thin and testable.
- **Consequences:** `pipelines/<name>/` is the home for future datasets (#10/#11 stamp out from it, inheriting the CI + Dataverse pattern). The Hub stays **L1**; pipelines are data-liberation sub-projects, not Hub level-climbs. Full per-extract provenance, day-resolution grain, and concept caveats (vertical-datum / capacity) are the pipeline's documented contracts. See `pipelines/reservoir-storage/AGENTS.md`.

## 2026-06-17 — Treat the source catalog as immutable raw, tracked in `data/raw/`

> **Superseded 2026-06-23** (see the entry above): the catalog now lives at `docs/planning/colorado_environmental_data_sources.json` and the top-level `data/` tree was retired. The decision below is kept as the historical record of why the catalog was version-controlled in the first place — that rationale still holds; only its location changed.

- **Context:** the curated catalog (`colorado_environmental_data_sources.json`) is small (~64 KB) and hand-compiled — more a curated source-of-record than a re-downloadable bulk file.
- **Decision:** place it in `data/raw/` and add a `.gitignore` carve-out so this one file is version-controlled, while the rest of `data/raw/*` stays ignored. New catalog vintages are new files, never in-place edits.
- **Why:** it is the project's actual tracked asset; versioning it gives a defensible history. The blanket "raw is git-ignored" convention assumes large/external inputs, which this is not.
- **Consequences:** the `context/` copy is retained as part of the design bundle; `data/raw/` is the canonical copy the data dictionary documents.

## 2026-06-17 — Frame the repo as the "Colorado Environmental Data Hub" (catalog now; corpus later)

- **Context:** the `context/` memos describe two layers — a source-inventory **catalog** (real data, present) and a journalist→article→citation **corpus** (specified as stubs in `architecture.md`).
- **Decision:** name the project the **Colorado Environmental Data Hub** (umbrella, matching the repo and the catalog JSON); document the catalog at L1 and record the corpus pipeline as the flagship L2+ roadmap initiative.
- **Why:** L1 documents what exists. The corpus is a future build; framing the Hub as the umbrella lets the catalog stand on its own while keeping the corpus discoverable in `ROADMAP.md`.
- **Consequences:** README/AGENTS center the catalog; the full pipeline lives in `ROADMAP.md` pointing at `context/architecture.md`.

## 2026-06-17 — Do not fire the full sensitive-data governance coupling at L1

- **Context:** the data present today is public agency metadata. The *planned* corpus will involve news article text (copyright) and journalist records (public figures acting professionally — low-risk per `context/methodology.md` §6.2), but neither is in the repo yet.
- **Decision:** keep L1 lean — no GOVERNANCE/responsible-data files now — but record the copyright/fair-use posture (metadata + excerpts + archived links, never republished full text) and the journalist correction/right-to-respond mechanism as **blocking** rows in `ROADMAP.md`.
- **Why:** generating governance scaffolds now would create dangling references and over-engineer a public catalog; dropping the duties silently would be worse. A named, blocking roadmap entry keeps the coupling honest without bloat. This is not a regulated/CARE/Indigenous-data case (which would force the coupling regardless of level).
- **Consequences:** before the corpus ingests article text or journalist PII, the project must climb to L3/L4 and satisfy those blocking items first.
